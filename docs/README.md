## 経歴

### FinTech 系スタートアップ（2024/06~現在）

【担当業務 1】ワークフローシステムの開発

各種申請や稟議などの業務手続きを管理するワークフローシステムの開発を担当。具体的には下記。

- Go/gRPC/Cloud spanner でマイクロサービスの設計、開発。複数のマイクロサービスが関与するシステムであったため、全サービスをローカル環境で起動するのではなく、開発時にはテストカバレッジを 100%担保。結合テストは Staging 環境で実施し、効率的に開発を進めた
- Typescript で BFF 開発。チーム内にサーバーサイド Typescript に知見のあるメンバーが少なかったためコードベースの改善提案やリファクタなどを先導し、開発体験を大幅に向上
- 外部会計システムとの連携機能を実装。Cloud pub/sub を使い非同期で処理させることでユーザーに待たせないような仕組みを実現
- 非同期処理の実行ステータスを管理するためのテーブルを作成。ステータスを可視化し、ログを追いやすくすることでトラブルシューティングを効率化した
- Spanner のストリーミング機能を利用し、CDC（Change Data Capture）の仕組みを実現。これにより、データ変更がリアルタイムで反映され、最新の状態を保持しやすくした

【担当業務 2】新規クレジットカードサービスの開発
法人カードのシステムの開発を担当。具体的には下記。

- マイナンバーカードを用いた本人確認を行うための機能を Go 言語/gRPC で実装。外部検証サービス（Verify API）のレスポンスがクセが強く、データの扱いに苦戦したが、必要な項目を適切にマッピングして対応した。
- 決済金額に応じてポイントを付与する仕組みを実装。ポイント管理を担う処理は今後の拡張性を考慮して新規マイクロサービスとして Go で実装し、他サービスから参照可能な形にした。ポイント付与を行う定期バッチを Kotlin で実装。Kotlin での開発はデビュー戦ながらもコルーチンを活用し、並列処理を活かしてポイント付与処理のスループットを向上させた。
- マイル交換機能を開発。航空会社からの「Excel レポートでの交換処理」という要件に合わせ、指定フォーマットへの自動変換バッチを定時実行する仕組みを構築し、手動オペレーションを大幅に削減。
- k8s/ArgoCD/GitOps を用いたインフラ環境構築。複数のマイクロサービスが頻繁に作成される環境で、Kubernetes、ArgoCD、GitOps を活用したインフラ環境の構築手順を整理。PR がマスターにマージされると、GCR にイメージがプッシュされ、デプロイ用リポジトリに PR が作成されるフローを整備した。また、ArgoCD を活用して、Staging 環境への自動反映を実現。新規サービスのデプロイ手順が複雑でドキュメント化されていなかったため、ドキュメントとしてまとめ他メンバーへの共有も行なった。

【発揮したバリュー】
Go 言語/gRPC を用いた開発では適切な粒度でマイクロサービスを分離するようにした。これにより、再利用性が向上し、変更や改善が他のサービスに波及しない形で実施できるようになり、メンテナンス性を大幅に改善した。Typescript のコードベースの改善を行い BFF の開発時間を大幅に短縮することに成功した。初挑戦であった Kotlin での開発には苦戦したものの素早くキャッチアップし、バグを生むことなく期限内にリリースすることに成功した。

### SalesTech 系スタートアップ（2023/11~現在）

【担当業務】チャットクライアントツールの開発。具体的には下記

- 技術選定。チームメンバーが Typescript の経験があることや外部 API との相性の良さから言語に Typescript を採用。フレームワークにはある程度雛形が定まっており情報も豊富にある Nestjs を採用した。
- Slack/Chatwork 連携機能の実装。メッセージやチャンネル情報を外部 API を叩いて定期バッチで取得し、システム内で活用できるようにした。外部 API はリクエスト回数に厳しい制限があるため sleep 処理をリクエスト間に導入することで制限に引っかからないように調整した
- Lambda 関数と AWS SES を使用して、メールの収集・処理システムを構築。SES で受信したメールを S3 バケットに保存し、その保存イベントをトリガーとして Lambda 関数が起動するようにした。Lambda 関数内で、メールの内容を解析し、必要な情報をデータベースに保存するフローを実現。
- Cognito を用いてアプリケーションに認証・認可機能を導入。JWT トークンを使用し、ユーザーが正しい資格情報を持っているかを確認する仕組みを構築。
- 手動作成したインフラの terraform 化
- テスト環境での実行速度と信頼性を向上させるために、Testcontainers を導入。実際のコンテナ化された環境でユニットテストや統合テストを実行できるようになり、本番環境に近い形でのテストが可能になった
- NestJS で構築したアプリケーションを GitHub Actions を利用して CI/CD パイプラインを作成し、AWS App Runner に自動デプロイする仕組みを実装

【発揮したバリュー】詳細仕様や開発体制など全く整っていない状況での参画であったが、他エンジニアと協業しながら技術選定やコーディング規約の作成、デプロイフローの整備など開発体制を一通り整えることに成功。要件が詰まっていない仕様はプロダクトオーナーと議論しながら仕様を具体化し、MVP 開発が円滑に進むようにドキュメント化を実施。

### FoodTech 系スタートアップ（2022/11~2024/04）

［担当業務 1］パンのサブスクリプションサービスの開発

- React/Typescript でのフロントエンド開発、Go 言語でのバックエンド開発、インフラの整備や CI/CD の構築などフルスタックに開発を担当
- パン出荷を通知するメール送信バッチの実装。Contentful 上に保存されているパンの写真を添付する必要があったが、Contentful API の実行制限に引っかからないようにマップを利用してキャッシュ化し、リクエスト回数減らすように工夫した
- Go で開発したアプリケーションを Cloud Run へデプロイ。Github Actions を用いて自動でデプロイできるよう CI/CD を構築した。デプロイ時に使用する Dockerfile はできるだけ軽量なものを使用するようにした

［担当業務 2］パン屋さん向け SaaS プロダクトの開発

- パン屋さんがパン出荷時に「パンと伝票のバーコードをスキャンできるようにする機能」の開発。gRPC の proto や GraphQL のスキーマ設計、API 実装、フロントエンド繋ぎ込みまで開発業務は一人で担当。
- Go/WebAssembly を用いたバーコード読み取り機能の実装。既存のバーコード読み取りライブラリではサポートされていないバーコード形式だったため自作した。
- 出荷伝票一覧を表示するため API 実装。N+1 問題のため初回読み込み時に時間がかかってしまうため、GraphQL の Dataloader を用いて回避するように実装。
- アジャイルチックにすすめビジネスサイドとパン屋さんに定期的にベータ版を提供し即座に開発にとり入れた
- 機能の開発を独りで担当したことから、属人化のリスクがあった。特にバーコードを中心とした複雑な実装や特定の仕様に関しては、適宜詳細なドキュメントを作成。さらに定期的に共有の時間を設け、情報をチームメンバーと共有することで、複数人が運用保守を行える体制を構築するよう努めた。

［担当業務 3］社内向け管理画面をローコードツールにリプレイス

- フルスクラッチで開発していた管理画面をローコードツールにリプレイス。プロジェクトリーダーを担当し、設計、開発からマネジメントまで担当
- 通信形式を GraphQL に統一。従来の管理画面では、Firestore、gRPC-Web、REST API が混在し複雑化していたため、BFF サーバーを導入し GraphQL に一本化。さらに、複数プロダクトを管理する構成を考慮し、各プロダクトに対応する gRPC API を新規実装。BFF はこれらの API を通じてデータを取得する設計に変更した。これにより、データ取得ロジックをプロダクト側に集約し、責務分離や保守性が向上。BFF から DB を直接操作する必要がなくなり、API の再利用性が高まり、システム全体のスケーラビリティと柔軟性が向上
- 業務委託メンバーのマネジメントを担当し、新規参画メンバーにはスムーズに業務を開始できるようオンボーディングを実施。タスク管理では、要件を明確かつ丁寧に記載し、開発負荷の軽減を図った。設計を自ら担当し、実装は業務委託メンバーに任せるなど、柔軟な対応で効率的なチーム運営を実現。
- 後輩エンジニアのマネジメントも行い、進捗を定期的に確認しながら適切にタスクを割り振った。エンジニア経験のないメンバーに対しては、不安や疑問に寄り添い、効果的なコミュニケーションを意識して指導を実施。トラブルが発生した際には迅速にサポートし、チーム全体のスムーズな開発を支えた
